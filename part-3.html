Debugging & Peer Reviews

Writing new code is only 30% of a professional developer's job. The other 70% is reading, debugging, and reviewing existing code. In this mid-term review, you will learn the industry standards for tracking down bugs and conducting professional Code Reviews.

1. The Art of Debugging in React

In Vanilla JavaScript, developers rely heavily on console.log() to figure out what went wrong. While logging is still useful, modern frontend engineering requires more powerful tools.

The industry standard is the React Developer Tools browser extension.

  • The Components Tab: Allows you to inspect the Virtual DOM, see exactly what Props a component is currently receiving, and view its active State without writing a single console.log.
  • The Profiler Tab: Helps you record interactions to see exactly which components are re-rendering too often and slowing down your application.
Reading the Red Screen of Death: When React crashes, it gives you a massive red error overlay. Do not panic. Always read the first line of the stack trace. 90% of the time, it tells you exactly which line of code caused the failure (e.g., "Cannot read properties of undefined").

2. The Top 3 Common React Bugs

As we hit the mid-term mark, let's review the three most common errors that crash junior developers' applications.

Mistake 1: The Infinite Loop 
// BAD: Executing the function immediately triggers a state change, 
// which causes a re-render, which clicks the button, which triggers state...
<button onClick={setCount(count + 1)}>Add</button>

// GOOD: Wrap it in an arrow function so it WAITS for the user to click.
<button onClick={() => setCount(count + 1)}>Add</button>
Mistake 2: Direct State Mutation 
const [user, setUser] = useState({ name: "Ali", age: 25 });

// BAD: React doesn't know you changed this, so the UI won't update!
user.age = 26; 

// GOOD: Always use the setter function!
setUser({ ...user, age: 26 });

3. Peer Code Reviews (Pull Requests)

In a professional environment, you cannot push your code directly to the live website. You must submit a Pull Request (PR) on GitHub or GitLab.

A Pull Request says: "Hey team, I built the new Cart feature. Can someone review my code before we merge it into the main project?"

Rules for Constructive Code Reviews:

  • Review the code, not the coder: Use "We" instead of "You". Say "We should probably rename this variable for clarity," instead of "You named this variable terribly."
  • Look for Architecture Flaws: Are they mutating state directly? Are they forgetting to use the key prop in a `.map()` loop?
  • Approve with LGTM: When the code is flawless, developers leave a comment saying LGTM (Looks Good To Me) and approve the merge.

Why this is essential for your career:

  1. Problem Solving is the Job: Seniority in software engineering is not determined by how fast you type, but by how quickly you can track down and fix a bug in a complex system.
  2. Team Collaboration: Bad code reviewers create toxic work environments. Learning how to leave kind, constructive, and highly technical feedback makes you a developer that companies actually want to hire and keep.
  3. Quality Assurance: Peer reviews act as the final net catching critical bugs before they reach the actual users (and cost the company money).
Industry Standard Tip: When stuck on a bug for more than 45 minutes, employ "Rubber Duck Debugging". Explain your code line-by-line out loud to an inanimate object (or a coworker). 80% of the time, hearing yourself say it reveals the logical flaw.
